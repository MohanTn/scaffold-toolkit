🏛️ Architecture — Scaffold CLI & Pack Integration with Claude Code Hooks & Skills

Deterministic, LLM-agnostic boilerplate scaffolding via CLI-driven template packs with Claude Code hook enforcement. · Node.js 20.12+ · TypeScript · ESM · Commander.js · Handlebars · Ajv

🤖 AI Overview — What Will Be Implemented

Scaffold-toolkit is a deterministic, LLM-agnostic CLI that renders boilerplate from template packs (DTOs, endpoints, services, route registration, frontend API clients) via intent manifests. A monorepo houses the core CLI (@mohantn/scaffold-core), domain-specific template packs in-repo (e.g., packages/templates-dotnet), and adapters for host LLM systems (Claude Code, GitHub Copilot). Claude Code integration ships four hooks—PreToolUse, UserPromptSubmit, PostToolUse, Stop—that enforce deterministic generation: the hard gate (PreToolUse) blocks direct writes to pack-owned files, forcing all boilerplate through scaffold generate; complementary safety nets (PostToolUse, Stop) catch unfilled AI_IMPLEMENTATION blocks at turn end. The pack system isolates template versions (e.g., v8-controller, v8-controller-gcp) as sibling folders, each with its own descriptor, test fixtures, and build-check script; new versions and cloud-provider extensions are additive, never colliding with existing registrations. Config (.scaffold/config.json) seeds packs by local path, project type, and pinned SHAs; scaffold init sets up the config, templates sync resolves packs to SHAs, and generate renders and injects against an intent manifest (JSON/TOON) that bridges the gap between natural-language user intent and deterministic boilerplate.

1 · Overview — Feature, Audience & Dependencies

Feature Summary

Scaffold-toolkit is a deterministic, LLM-agnostic CLI that generates boilerplate code (DTOs, endpoints, validators, database layers) from template packs driven by intent manifests. It eliminates latency and drift by splitting code generation into deterministic (file rendering, marker injection, hashing in CLI) and probabilistic (translating intent, implementing logic in LLM) halves. A monorepo houses the core CLI, domain-specific template packs (.NET, Node.js), and host adapters (Claude Code, GitHub Copilot) that enforce deterministic generation via hooks.

3 · C4 — Context & Containers

C4 Context: Landscape

Scaffold sits at the intersection of three actors: Developer (end-user, fills manifest intent and AI_IMPLEMENTATION blocks), Coding Agent (LLM) (translates natural-language request into manifest, implements join logic), and Template Pack (defines boilerplate templates, descriptors, test fixtures). npm Registry publishes @mohantn/scaffold-core and @mohantn/scaffold-adapter-claude-code. Claude Code Runtime executes hooks to enforce deterministic generation (PreToolUse gates direct writes, PostToolUse/Stop nudge at turn end).

C4 Container: Internal Architecture

CLI (Commander): Exposes commands (init, generate, undo, status, bootstrap-markers, validate-pack, check-edit). Config Loader: Reads/writes .scaffold/config.json (project type, packs, pinnedSha). Template Sync: Git clone, SHA pin, descriptor parsing, version catalog. Manifest Schema: Ajv validation, entity/fields/options, per-pack schemas. Handlebars Render: Entity, fields, options, custom helpers, file write to disk (dry-run mode). Marker Injection: SCAFFOLD_START/END, AI_IMPLEMENTATION blocks, hash trail, idempotency, creationGate checks. Changeset Audit: SHA256 hashing, pre/post state, undo/rollback logic, .scaffold/changesets.json. Claude Code Hooks: PreToolUse (blocks direct writes), UserPromptSubmit (injects context), PostToolUse (nudges on generate), Stop (verifies filled blocks).

Data Flow: Manifest to Output

User's request (natural language) → Coding Agent (LLM) → Build Intent Manifest (JSON/TOON with targetStack, entity, fields[], options) → CLI: scaffold generate --manifest <file> → Load Config (.scaffold/config.json) → Load Pack Descriptor → Ajv Validate Manifest Against Pack Schema → For each .hbs template in pack/version/: Render with (entity, fields[], options, pack context), Write to disk (or dry-run preview) → For each SCAFFOLD_START/END marker in target files: Inject rendered boilerplate or AI_IMPLEMENTATION block, Hash pre/post state for audit trail → Write Changeset (SHA256, file list, undo info) → Output Report (TOON/JSON with generated, modified, skipped, empty_implementations) → Coding Agent reads report → Agent fills AI_IMPLEMENTATION blocks using Edit tool → Claude Code PreToolUse hook intercepts Edit → Hook runs: scaffold check-edit --file X --tool edit → If file is inside AI_IMPLEMENTATION block: allow; If file is in pack-owned marker (but outside impl): block → File write proceeds or fails.

4 · Domain & Data Model

Domain Model

Manifest: { manifestSchemaVersion (int), targetStack (string), entity (string), fields (array of {name, type}), options (object) }. Config: { projectType (string), packs (map of name → PackConfig) }. PackConfig: { path (string), version (string), pinnedSha (string) }. PackDescriptor: { schemaVersion (int), name (string), version (string), displayName (string), injectionMarkers (object), aiImplementationBlockNames (object) }. Changeset: { id (string), timestamp (string), manifestSha (string), fileHashes (object), generated (array of File), modified (array of File) }. File: { path (string), preSha (string), postSha (string), status (string) }.

Schema Changes & Migrations

Status: No database schema migrations. All state is file-based (.scaffold/config.json, .scaffold/changesets.json, per-repo). Manifest Schema Versioning: The manifest schema is versioned via the manifestSchemaVersion field (currently: 1). If the schema changes, new manifests must set the new version; the CLI validates against the pack's actual schema at generate time. Backward compatibility is the pack author's responsibility (e.g., if v8-controller adds a new required field, a new manifest schema version is introduced). Pack Descriptor Versioning: Each template pack version (e.g., v8-controller, v8-controller-gcp) has its own descriptor.json defining injectionMarkers, aiImplementationBlockNames, and schema. Non-colliding marker names (e.g., SCAFFOLD_V8_CONTROLLER_END vs. SCAFFOLD_V8_CONTROLLER_GCP_END) ensure multiple versions can coexist in the same repo.

6 · UI/UX (React)

Developer Workflow

1. New Repo: No .scaffold/ yet. 2. Init: Run scaffold init with --project-type and --pack specs. 3. Sync Packs: Run scaffold templates sync to pin SHAs. 4. User Request: "Add Invoice endpoint". 5. Manifest: Coding Agent builds intent JSON. 6. Generate: Run scaffold generate --manifest Invoice.json. 7. Report: CLI outputs generated files, empty blocks. 8. Fill Impl: Coding Agent fills AI_IMPLEMENTATION blocks using Edit tool. 9. PreToolUse: Hook gates file edits, allows inside impl blocks. 10. Status Check: Stop hook runs scaffold status --json, verifies all blocks filled. Turn ends.

Component Hierarchy & API Integration

App (scaffold CLI): Commander Router (init, manifest, templates, generate, undo, status, bootstrap-markers, validate-pack, check-edit commands) → Config/Pack/Manifest processing → Handlebars Render + Marker Injection + Changeset Audit → Report Output. Hook Adapters (Claude Code): PreToolUse (blocks direct writes), UserPromptSubmit (injects context), PostToolUse (nudges on generate), Stop (verifies filled blocks). Pack System (external, loaded at runtime): Descriptor (injectionMarkers, schemas), Templates (Handlebars .hbs files), Test Fixtures (test_data/), Build Check Script (tools/validate-build.mjs).

7 · Behaviour — State & Process

Entity State Lifecycle

[*] → Undefined → Configured (scaffold init) → Ready (templates sync) → GenerationPending (User intent, Agent builds manifest) → Generated (scaffold generate) → Implementing (Report lists empty AI_IMPLEMENTATION blocks) → Implementing (Agent fills blocks via Edit, PreToolUse gates) → Filled (All blocks completed) → [*] (Turn ends, Stop hook verifies). Also: Generated → Undone (scaffold undo <changeset-id>) → Ready (Changeset rolled back). Also: Ready → Validated (scaffold validate-pack) → Ready (Smoke-test passes) or Error (Smoke-test fails, pack author fixes).

Command Execution Flow

scaffold generate --manifest Invoice.json: Load args → Load .scaffold/config.json → Parse targetStack from manifest → Lookup pack in config by name (Not found? Exit with error) → Get pinnedSha for pack (pinnedSha not set? Tell user to run templates sync) → Load pack descriptor from cache → Ajv validate manifest against pack schema (Invalid? Exit with validation error) → Load pack's Handlebars templates → For each template file in pack/version/: Render with { entity, fields, options, pack context }, Write to target file (or preview in dry-run), Record hash (pre: null or prior, post: SHA256) → For each SCAFFOLD_*_START/END marker in modified files: Check if marker content differs from prior inject (hash trail). If differs and !--force, exit (user hand-edits detected). Inject marker-wrapped boilerplate or AI_IMPLEMENTATION_START/END block. Record hash (pre: prior, post: new) → If creationGate is overwrite-required: Check if --force flag provided. If !--force, exit with message "use --force to overwrite" → Write .scaffold/changesets.json (audit trail) → Generate report (list generated, modified, empty AI_IMPLEMENTATION blocks, skipped) → Output report (TOON or JSON) → Exit code 0 (success).

Error Handling & Retry Logic

Manifest validation fails: Clear ajv error message, show which field is invalid, suggest manifest new to rebuild, Exit code 1. Pack not found in config: Name the configured packs, suggest scaffold init --pack, Exit code 1. Pack pinned SHA not set: Suggest templates sync, Exit code 1. Template file not found: Name the missing file, check pack version/descriptor, Exit code 1. Marker content differs: Show file + line range, explain user hand-edits detected, suggest --force to override, Exit code 1. Handlebars render error: Show template + error context, check manifest field types, Exit code 1. File locked (Windows): Suggest close file in IDE, Retry after brief delay (future), Exit code 1. Disk full: OS-level error propagated, Exit code 1. PreToolUse hook blocks Edit: Hook output: reason + file path, suggest running generate first, Tool call fails (Claude Code rejects).

8 · Deployment — Docker, Terraform, GitLab CI/CD

Deployment Architecture (npm, monorepo)

Packages: scaffold-toolkit (private monorepo) — Root package.json, @mohantn/scaffold-core (packages/core, CLI TypeScript ESM, published to npm), @mohantn/scaffold-adapter-claude-code (packages/adapter-claude-code, Hooks + Skill.md, published to npm), Template Packs (packages/templates-dotnet, packages/templates-node, NOT published, vendored in-repo). CI/CD Pipeline (.github/workflows/ci.yml): Step 1: build-test-lint (npm ci, build, lint, test all workspaces). Step 2: templates-dotnet-build-check (validate-build.mjs script, real dotnet build on test fixtures). Step 3: templates-node-build-check (validate-build.mjs script, real npm build on test fixtures). Step 4: publish to npm (only on main merge, requires all prior jobs to pass). Package Distribution: @mohantn/scaffold-core: published to npm, audience: end-users, agents via npx. @mohantn/scaffold-adapter-claude-code: published to npm (private: true), audience: Claude Code skill install location, hooks/ directory. Templates (dotnet, node): not published, in-repo only, vendored from external standalone repos.

Terraform / Infrastructure

Status: No server-side infrastructure. Scaffold is a local CLI tool; no deployments, no cloud resources, no databases. All state is file-based (.scaffold/config.json, changesets.json) stored in target repos.

Incident Response Flow

Template pack broken (render error): Detected: scaffold generate fails. Owner: Pack author notified via issue. Mitigation: Use prior pack version or --pack-version. Hook not installed or outdated: Detected: User tries Write/Edit without check-edit gating. Owner: Skill installer (Claude Code). Mitigation: Re-run hook install step from SKILL.md. Incompatible manifest/pack versions: Detected: Ajv schema mismatch. Owner: Agent + pack author (schema drift). Mitigation: Rebuild manifest with manifest new, or update pack descriptor. AI_IMPLEMENTATION blocks not filled: Detected: Stop hook runs status check at turn end. Owner: Agent (missed blocks). Mitigation: Hook nudge; agent retries fill. Unexpected manual edits in markers: Detected: Hash trail differs. Owner: User (conflicting hand-edits outside impl). Mitigation: --force flag to override, or manual merge. CI pack build-check flakes: Detected: templates-dotnet-build-check job fails. Owner: Pack author (template compilation error). Mitigation: Fix template, push, CI re-runs. Template pack packs get stale: Detected: No updates for months. Owner: Pack author. Mitigation: Packs are vendored in this repo; updates are pull requests. Standalone repos (scaffold-templates-dotnet) are the original source of truth.
