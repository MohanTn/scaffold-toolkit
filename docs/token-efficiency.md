# Token efficiency: LLM output only where thinking is needed

Goal: for an enterprise .NET Core project, reduce LLM-generated output to the probabilistic code that genuinely needs reasoning. Everything deterministic renders from the template pack instead.

## Suggestions, sorted by ROI (high → low)

1. **Manifest-first workflow** (implemented, see below). For any new entity/endpoint the LLM emits only a compact spec; `scaffold generate` expands it into the full Clean Architecture stack (~25 files: controller, commands, queries, handlers, DTOs, repository, EF configuration, csproj, tests). ~95% token cut per entity.
2. **LLM writes only rule bodies** (implemented, see below). Restrict probabilistic output to validator rules, domain methods, and handler business logic. Scaffold everything else, then hand the LLM only those files or marked regions to fill.
3. **Injection markers over LLM edits** (implemented, see below). Never let the LLM edit `Program.cs`, DI registration, or `DbContext` for wiring; `dbset-registration` / `di-registration` markers inject deterministically. Saves both the edit tokens and the re-read tokens of large shared files.
4. **Test skeletons from templates** (implemented, see below). Handler test templates render the arrange/act boilerplate; the LLM adds only assertions for the business rules it wrote.
5. **Config variants via manifest `options`, not regeneration** (implemented, see below). Provider or environment swaps (e.g. `options.database.provider`) are a one-line manifest change instead of LLM-rewritten templates.
6. **Gap-fill the pack once, save forever** (implemented, see below). Any file type the LLM writes twice per entity (mapping profiles, pagination queries, integration-test fixtures) becomes a new `.hbs` template. One-time template cost, zero marginal tokens after.
7. **Architectural additions as sibling version folders** (implemented, see below) (e.g. `v8-controller-gcp`), layered at generate time, so new targets reuse the base templates instead of the LLM re-emitting them.
8. **Build-gate to kill retry burn** (implemented, see below). `validate-build.mjs` in CI catches namespace/using/compile errors deterministically; without it, failures come back to the LLM as expensive regenerate-and-retry loops.
9. **Measure with the benchmark harness** (`npm run benchmark`) to confirm which changes actually cut tokens before investing in more templates.

### Stack-agnostic engine: Node.js reference pack

The suggestions above were pioneered with `packages/templates-dotnet/v8-controller` (.NET). To prove the same token-efficiency strategies apply across languages and architectures, `packages/templates-node/generic-v1` is a non-.NET reference pack for Node.js/TypeScript feature modules. It demonstrates: pack-declared `inputs[]` contract (a single `name` string instead of the dotnet pack's `entity` + `fields`), pack-level `commentSyntax` override (/// for .ts files, not //, so the override is proven in generated bytes), injection with `append` strategy (module registry grows on each generation), and the same required `AI_IMPLEMENTATION` blocks for business logic. Use it as a template for authoring a pack for any stack:

```sh
scaffold manifest new --stack module --input name=Billing
scaffold generate --manifest billing.manifest.json
```

## Manifest-first workflow (suggestion 1)

The LLM never writes generated code, and does not even hand-write the manifest JSON. It emits one command line:

```sh
scaffold manifest new \
  --stack backend \
  --entity Invoice \
  --field Amount:decimal \
  --field IssuedAt:DateTime \
  --option rootNamespace=Acme.Billing \
  --option database.provider=sqlite \
  --out invoice.manifest.json

scaffold generate --manifest invoice.manifest.json
```

`manifest new` expands the spec into a schema-validated intent manifest (`--field name:type` pairs; `--option` dot-paths nest, `true`/`false`/numbers coerce). `generate --dry-run` previews the file plan without writing. After generation, the LLM's remaining job is suggestion 2: fill in the rule bodies.

## Rule-body regions and frozen wiring (suggestions 2 and 3)

Every place in the `v8-controller` pack where code genuinely needs thinking carries a `SCAFFOLD:AI_IMPLEMENTATION:START/END` region: command/query handlers, handler unit tests, `Create`/`Update` validators (rules), and the domain entity (invariants, factory methods, state transitions). `scaffold check-edit` — which the host adapters' `PreToolUse` hooks call before any Write/Edit runs — allows edits only inside those interiors:

- Pack-owned file writes are blocked; creation goes through `scaffold generate`.
- Wiring surfaces (`Program.cs` DI, `ApplicationDbContext` DbSets, `DependencyInjection.cs` repository registrations) are `SCAFFOLD:<marker>` injection regions, filled per-entity by `generate`, never by LLM edits.
- Edits inside an `AI_IMPLEMENTATION` interior (a validator rule, a handler body, the domain-behavior region) are allowed.

`packages/templates-dotnet/tools/check-guardrails.mjs` asserts this contract against the real CLI in a throwaway project (`npm run check:guardrails:templates-dotnet`) and runs in CI ahead of the dotnet build-check.

## Test skeletons (suggestion 4)

Handler test templates render the full arrange/act deterministically: mocked repository and unit of work, handler construction, command/query construction from the manifest's fields, the `Handle` call, and the not-found scenario. Only the assertion block sits in an `AI_IMPLEMENTATION` region (pre-filled with working default assertions), so the LLM's test-writing job is reduced to asserting the business rules it implemented. `check-guardrails.mjs` proves arrange/act is frozen and assertions are editable.

## Option variants and the pack gap-fill (suggestions 5 and 6)

`options.database.provider` switches the EF Core provider (`postgres` → Npgsql, default → Sqlite) across `InfrastructureCsproj` and `InfrastructureDependencyInjection` from one manifest line. `test_data/variants/database-provider-postgres.json` builds in its own sample project inside `validate-build.mjs`, so the non-default branch is compiled in CI, not just rendered. Gap-fill example shipped with this doc: `Create`/`Update` validator unit tests were LLM-written every entity; they are now `CreateEntityCommandValidatorTests.cs.hbs` / `UpdateEntityCommandValidatorTests.cs.hbs` targets rendered per entity.

## Sibling version layer (suggestion 7)

`v8-controller-gcp` is an additive sibling of `v8-controller`: its own descriptor (`inputs` requires only `options`, no entity), its own templates (Dockerfile, `cloudbuild.yaml`, `appsettings.Production.json`), zero injections, so no marker collisions with the base pack. It is configured as a second pack slot and layered onto a scaffolded project:

```sh
scaffold init --project-type dotnet \
  --pack backend=packages/templates-dotnet@v8-controller \
  --pack backend-gcp=packages/templates-dotnet@v8-controller-gcp
scaffold generate --manifest entity.manifest.json      # base stack
scaffold generate --manifest gcp.manifest.json         # targetStack backend-gcp
```

## Build-gate coverage (suggestion 8)

`validate-build.mjs` (CI job `templates-dotnet-build-check`, in the `publish` job's `needs:` list) now covers all of the above: 24 entities plus the new test skeletons through `dotnet build` and `dotnet test`, the GCP layer's file outputs, and each `test_data/variants/*.json` scenario compiled in an isolated sample project.
