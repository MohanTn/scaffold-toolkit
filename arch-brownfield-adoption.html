🧠 Shared Understanding — Brownfield Adoption for Existing, Unmarked, Non-Standard-Path Services

A single unified command (extending scaffold bootstrap-markers) that scans an existing codebase, auto-generates a path mapping from the pack's declared targets/injections to real repo files, persists it in .scaffold/config.json, inserts markers, and makes the enforcement hooks mapping-aware. · scaffold-toolkit · packages/core (TypeScript ESM) · packages/adapter-claude-code hooks · packages/templates-dotnet

🤖 At a Glance — Do We Agree on the Goal?

My understanding of what you want

Approved. One unified command (extending scaffold bootstrap-markers in place, not a second parallel scaffold adopt) scans a brownfield repo, and for each entry a synced pack descriptor already declares in targets[].output/injections[].file, resolves and persists a mapping to the real file in .scaffold/config.json per pack slot. Confident matches are written immediately (mapping + empty SCAFFOLD:<marker> pairs); ambiguous ones are left flagged/unmapped in the same config file for the user to fill in by hand — no blocking per-file confirmation prompt. Path-layout differences (e.g. Services/ vs. the pack's hardcoded convention) are solved the same way: a resolved pathConfig/companyProjectName block persisted per pack slot, read automatically by every subsequent generate and check-edit call, replacing the old per-manifest-only convention. collectPackOwnership.ts and the PreToolUse/PostToolUse/Stop hooks now consult that persisted mapping, so an adopted file is gated identically to a freshly scaffold-generated one. Where the mapper can't confidently place a marker inside a resolved file (no cataloged anchor, no pack-declared bootstrapAnchors[]), the command refuses and reports needsManual rather than guessing — never a speculative whole-file wrap. This ships CLI-only: the user runs it themselves, with no wiring into the Claude Code adapter's SKILL.md agent-driven workflow.

1 · The Concept, In My Own Words

Right now, scaffold generate assumes a greenfield or already-scaffolded target: every file it writes into is either brand new, or an existing file that scaffold itself put markers into on a prior run. The moment you point it at a real brownfield repo — one with an OrdersController.cs that a human wrote by hand two years ago, no markers, possibly not even matching the pack's expected directory layout — two independent things break at once: (1) the injector can't find the marker it's looking for and aborts the whole run with zero writes, and (2) even if markers existed, check-edit's ownership detection is a pure path-pattern match against the pack's hardcoded template, so a file at the wrong path is invisible to the ownership system no matter what's inside it.

The approved design is deliberately a single command: scaffold bootstrap-markers is extended in place to absorb both problems (mapping real files to the pack's declared targets/injections, and inserting the markers those files need) rather than adding a second, parallel scaffold adopt. It writes confident matches immediately and never guesses on ambiguous ones — a file with no clear anchor is reported needsManual, full stop, matching the existing bootstrap-markers philosophy.

2 · What the Code Does Today

Two mechanisms exist today, and neither solves brownfield adoption in general:

Ownership detection (packages/core/src/checkEdit/collectPackOwnership.ts:32-71) reverse-compiles every configured pack's targets[].output and injections[].file Handlebars path templates into regexes, and tests candidate paths against them on every check-edit call. There is no persisted "here are the files we own" list — ownership is recomputed from the pack descriptor every time. A file whose real path doesn't match the template (wrong directory convention) is simply invisible to this system: packOwned: false, same as a file the pack has nothing to do with.

Marker bootstrapping (packages/core/src/bootstrapMarkers/bootstrapMarkers.ts) is the one real brownfield primitive that exists. It walks a fixed catalog of 4 dotnet pack versions (anchorCatalog.ts:91-104), each mapped to a handful of specific filenames (Program.cs, AppDbContext.cs, a DI extensions file), finds a single unambiguous anchor point via regex or brace-matching, and splices in empty marker pairs — never marker content. It refuses to touch dirty or untracked files (gitSafety.ts:49-61) and reports ambiguous matches as needsManual rather than guessing. This is exactly the git-safety and anchor-placement machinery the approved unified command reuses and extends.

flowchart TD
  A[Existing repo file] --> B{In ANCHOR_CATALOG\nfor this pack version?}
  B -- No --> C[unsupportedPacks: skipped entirely]
  B -- Yes --> D{Git-tracked and clean?}
  D -- No --> E[needsManual: untouched]
  D -- Yes --> F{Exactly one anchor match?}
  F -- No 0 or 2+ --> E
  F -- Yes --> G[Insert empty SCAFFOLD marker pair]

7 · Scope Boundaries

In scope:

  • scaffold bootstrap-markers extended in place (not a second, parallel command) to both map existing files to the pack's declared targets[].output/injections[].file entries and insert markers into them
  • Auto-generating and persisting that mapping into .scaffold/config.json per pack slot, editable by the user at any time afterward; confident matches written immediately, ambiguous ones left flagged
  • A persisted pathConfig/companyProjectName block per pack slot in .scaffold/config.json, replacing the old per-manifest-only convention
  • Updating collectPackOwnership.ts and the PreToolUse/PostToolUse/Stop hooks to consult the persisted mapping, so an adopted file is gated identically to a freshly scaffold-generated one
  • Refuse-not-guess as the fixed policy when no confident anchor exists for a marker (no whole-file-wrap fallback, ever)

Out of scope:

  • Changing scaffold generate's existing all-or-nothing failure semantics for already-adopted repos (T2) — that contract stays as-is
  • Normalizing, refactoring, or rewriting the business logic inside adopted files (U10) — adoption is structural registration only
  • Any non-dotnet pack's specific adoption story beyond making the underlying engine mechanism pack-agnostic (U12) — templates-react or a hypothetical new pack would still need its own bootstrapAnchors[] declarations
  • Extending the SKILL.md agent-driven workflow to trigger adoption mid-conversation (OQ4, resolved: out of scope) — CLI-only, user-run
