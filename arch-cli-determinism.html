1 · The Concept, In My Own Words

The Concept, In My Own Words

Your goal is to make sure the hooks force determinism—not trust the AI to follow instructions. When Claude Code or Copilot is configured with scaffold hooks, they cannot bypass scaffold generate and hand-write boilerplate, no matter what the prompt says. The hooks intercept before writes happen and say: "Is this a pack-owned file? If yes, use generate or I'll block you."

Why this matters: AI sometimes ignores skill instructions and does what seems right to it (write the file directly). Hooks make it structurally impossible. That's the difference between "please use generate" (hope) and "you cannot bypass generate" (guaranteed).

Scope: a single invocation of scaffold generate. Given the same manifest and repo state, every provider produces identical outputs. Multi-turn coherence, agent latency, prompt variation—those are out-of-scope.

2 · What the Code Does Today

What the Code Does Today

The scaffold CLI has three parts: a pure orchestrator (scaffold generate), structural gates (scaffold check-edit, scaffold status), and hook wiring (adapter-specific).

The core orchestration:

When you run scaffold generate --manifest <file>:

  1. Load the manifest (.json or .toon)
  2. Load the pack descriptor (schema, targets, injections)
  3. Render templates using Handlebars + manifest data
  4. For each target file: create new or inject into existing
  5. For each injection point: scan existing file for the marker, inject boilerplate between SCAFFOLD markers
  6. Write changes to disk (or just report with --dry-run)
  7. Record file hashes and pending AI_IMPLEMENTATION blocks
  8. Print a TOON report listing what happened

This is deterministic: same manifest + same repo state = identical outputs. Crucially, --dry-run uses the same code path as the real run — only the final disk writes are gated on !dryRun. This guarantees the dry-run report matches reality if nothing changes in between.

Structural gates:

scaffold check-edit --file <path> --tool <write|edit> is the universal gate both adapters invoke before permitting file changes. It asks: "Is this file pack-owned? If so, is this edit allowed?" The answer is purely structural: depends only on .scaffold/config.json + pack descriptor + file content.

scaffold status scans all AI_IMPLEMENTATION blocks and reports which ones are still unfilled.

How hooks enforce determinism:

Claude Code: Pre-Tool-Use hook runs scaffold check-edit before Write/Edit tool calls execute, blocking pack-owned writes immediately. Stop hook runs scaffold status at turn-end, blocking if any blocks are unfilled.

Copilot CLI: Same two hooks (registered in .github/hooks/scaffold-toolkit.json), plus a precheck in the gh scaffold generate wrapper that refuses to start a new generate if prior blocks are unfilled. Payload format for Copilot's hook is verified: { timestamp, cwd, toolName, toolArgs }.

Both adapters use the same check-edit and status logic — they just invoke it through different hook mechanisms. This is the foundation of determinism.

7 · Scope Boundaries

Scope Boundaries

In Scope (v2):

  • Claude Code and Copilot CLI adapters enforce determinism via hooks
  • CLI exit codes, file outputs, and report structure are identical across Claude and Copilot for the same manifest + repo state
  • Dry-run behavior matches real-run behavior for identical working-tree state
  • Structural gates (check-edit, status) are invoked uniformly via hooks (PreToolUse, Stop/agentStop)
  • Hash-trail idempotency: scaffolding the same entity twice doesn't corrupt files
  • Hooks are mandatory: they fire before writes and block rule violations

Out of Scope (v2):

  • Agent behavior: Whether Claude Code and Copilot produce the same prompts, latency, or multi-turn coherence
  • Pi Agent: Deferred to future release. Pi has hooks (verified), but not in the v2 delivery scope
  • Manual CLI invocation: If someone runs scaffold generate by hand outside hooks, we don't enforce the rules. Hooks are optional; determinism is guaranteed, but enforcement isn't
  • Environment divergence: Assume uniform context (same PATH, Node ≥20.12, working directory). Setup docs enforce this
  • Pack content mutations: If a pack changes between invocations, output changes. Assume pack cache is stable
  • Auto-install hooks: One-time explicit setup per repo. Auto-detection deferred to future
