1 · Overview — Feature, Audience & Dependencies

Feature Summary

Today, enforcement that a host LLM uses scaffold generate instead of hand-writing boilerplate is too weak and too late: PostToolUse only fires on Bash commands containing scaffold generate, and Stop/agentStop only block on unresolved .scaffold/pending/*.json records that only exist if generate already ran. If the host LLM never calls generate at all and just hand-writes a controller, nothing today stops it. This feature adds a new scaffold check-edit command and PreToolUse hooks on both host adapters that structurally block any Write/Edit touching a pack-owned file outside an AI_IMPLEMENTATION marker interior, whenever .scaffold/config.json is present — and are a complete no-op, zero added cost, when it is not. A complementary UserPromptSubmit hook on Claude Code (Copilot's equivalent event is notification-only and cannot inject context) states the rule up front every turn. A new reusable tools/benchmark/ harness measures the token/time/build-quality delta between freehand and CLI-scaffolded authoring of the same feature, on demand.

Assumptions

  • .scaffold/config.json presence is a reliable, cheap signal that a repo wants enforcement — no content/keyword-based intent detection is used, per explicit decision.
  • Pack descriptors are trustworthy at check time: they come from a pinned, already-synced SHA in the local cache, not arbitrary live network content.
  • Claude Code's PreToolUse supports a real block/deny decision (confirmed for the general Claude Code hook family; the exact JSON shape for this specific event was not independently re-verified this session — see Section 10).
  • GitHub Copilot CLI's preToolUse supports modify/block (confirmed to exist via the hooks reference; its exact input field names for file write/edit tool calls were not confirmed — see Section 10, this is the plan's single highest-risk open item).

Out of Scope

  • Auto-migrating existing hand-written boilerplate into pack-owned marker regions.
  • A repo-wide hard gate on every file — deliberately narrowed to pack-owned target/injection paths only, per explicit user decision, so unrelated docs/tests/business-logic edits are never affected.
  • Actually running the benchmark harness for real and publishing a results report — this feature ships the harness as a reusable tool, not a one-time results artifact.
  • Cleaning up dotnet_folder/ at the repo root, confirmed during research to be dead scratch state pointing at a nonexistent /tmp/dotnet-scaffold-test/pack path — flagged to the user separately, not touched by this feature.

Domain Model

classDiagram

    +Map~string,PackEntry~ packs
  }
  class PackEntry {
    +string url
    +string version
    +string pinnedSha
  }
  class Descriptor {
    +TargetSpec[] targets
    +InjectionSpec[] injections
  }
  class TargetSpec {
    +string output
    +string mode
  }
  class InjectionSpec {
    +string file
    +string marker
    +string strategy
  }
  class AiImplementationBlock {
    +number startLine
    +number endLine
    +string content
    +boolean empty
    +boolean required
    +number interiorStartOffset
    +number interiorEndOffset
  }
  class CheckEditRequest {
    +string tool
    +string file
    +string oldString
  }
  class CheckEditResult {
    +string decision
    +string reason
    +boolean packOwned
    +MatchedOwner[] matchedBy
  }
  PackConfig "1" --> "*" PackEntry
  PackEntry --> Descriptor : resolves via local cache
  Descriptor "1" --> "*" TargetSpec
  Descriptor "1" --> "*" InjectionSpec
  CheckEditRequest --> CheckEditResult : evaluated against
  CheckEditResult --> AiImplementationBlock : edit containment check

interiorStartOffset and interiorEndOffset are new, additive fields on AiImplementationBlock — markerScan.ts's MarkerLocation type already computes the equivalent offsets for SCAFFOLD:<marker> pairs; this feature applies the same buildLineOffsets technique to AI_IMPLEMENTATION blocks.

Entity-Relationship / Data Store

None. This feature introduces no persistent data store and no schema. All state it reads is either (a) the target repo's own existing files and .scaffold/*.json records, already produced by the existing generate/status commands, or (b) the benchmark harness's own ephemeral tools/benchmark/results/<timestamp>/*.json output, which is gitignored and not modeled as durable state.

Migration Impact

None required. The only change to an existing type is additive: two new optional-in-practice fields on AiImplementationBlock, populated by scanAiImplementationBlocks the same way MarkerLocation already populates its own offset fields. Existing consumers (status.ts, generate.ts's report builder) read only the fields they already use, so this is source- and behaviour-compatible.

CLI & Hook Interface Spec (not an HTTP API — this feature has no network surface)
command: scaffold check-edit
arguments:
  --file:            { type: path, required: true }
  --tool:            { type: enum[write, edit], required: true }
  --old-string:      { type: string, required: if tool == edit, mutually_exclusive_with: --old-string-file }
  --old-string-file: { type: path,   required: if tool == edit and old-string too large, mutually_exclusive_with: --old-string }
exit_codes:
  0: allow
  1: block
stdout_shape:
  decision:   "allow" | "block"
  reason:     string, present iff decision == "block"
  packOwned:  boolean
  matchedBy:  [{ pack: string, kind: "target" | "injection", pattern: string }]

Gate Decision Sequence

sequenceDiagram
  participant Host as Host Agent (Claude/Copilot)
  participant Hook as PreToolUse Hook Script
  participant CLI as scaffold check-edit
  participant Cache as Local Pack Cache
  participant Marker as markerScan.ts
  Host->>Hook: attempts Write/Edit on file X
  Hook->>Hook: existsSync(.scaffold/config.json)?
  alt config absent
    Hook-->>Host: {} (no-op, allow)
  else config present
    Hook->>CLI: execFileSync scaffold check-edit --file X --tool ...
    CLI->>Cache: load synced pack descriptor(s)
    CLI->>CLI: templateToRegex match against targets[]/injections[]
    alt not pack-owned
      CLI-->>Hook: {decision: allow, packOwned: false}
    else pack-owned and tool == write
      CLI-->>Hook: {decision: block, reason: "...run scaffold generate"}
    else pack-owned and tool == edit
      CLI->>Marker: scanAiImplementationBlocks(content)
      Marker-->>CLI: interior offset ranges
      CLI->>CLI: is old_string fully inside an interior?
      CLI-->>Hook: {decision: allow} or {decision: block}
    end
    Hook-->>Host: deny (blocked) or allow, translated to host's own shape
  end

7 · Behaviour — State & Process

Pack-Owned File Lifecycle

stateDiagram-v2
  [*] --> Unrendered
  Unrendered --> Rendered: scaffold generate (targets/injections written, markers present)
  Rendered --> Implemented: Edit inside AI_IMPLEMENTATION interior
  Implemented --> Implemented: further Edit inside same interior
  Unrendered --> Unrendered: Write attempt blocked (not yet rendered)
  Rendered --> Rendered: Write attempt blocked (already rendered, still pack-owned)
  Rendered --> Rendered: Edit outside interior blocked (injection region or other content)
  Implemented --> Implemented: Edit outside interior blocked

Note: every state's transition into a blocked self-loop is structural, not session-tracked — the gate never needs to know whether generate "has run this turn"; it only asks whether the specific edit's target text falls inside a currently-scanned AI_IMPLEMENTATION interior.

check-edit Decision Activity

flowchart TD
  Start([check-edit invoked]) --> ConfigCheck{.scaffold/config.json exists?}
  ConfigCheck -->|no| AllowNoConfig[[allow: packOwned=false]]
  ConfigCheck -->|yes| PathCheck{path inside repo root?}
  PathCheck -->|no| AllowOutside[[allow: not matchable]]
  PathCheck -->|yes| Matchers[Build pack-owned matchers from synced descriptors]
  Matchers --> MatchTest{path matches a target/injection pattern?}
  MatchTest -->|no| AllowNotOwned[[allow: packOwned=false]]
  MatchTest -->|yes| ToolCheck{tool == write or edit?}
  ToolCheck -->|write| BlockWrite[[block: must use generate]]
  ToolCheck -->|edit| FileExists{file exists on disk?}
  FileExists -->|no| BlockMissing[[block: cannot verify interior of nonexistent content]]
  FileExists -->|yes| FindOld{old_string found exactly once?}
  FindOld -->|no / ambiguous| BlockAmbiguous[[block: fail-closed, ambiguous]]
  FindOld -->|yes| Interior{fully inside an AI_IMPLEMENTATION interior?}
  Interior -->|yes| AllowInterior[[allow]]
  Interior -->|no| BlockOutside[[block: outside interior / in injection region]]
