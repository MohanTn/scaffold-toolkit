1 · The Concept, In My Own Words

Approved. F1 (scaffold next) and F5 (scaffold pack new) are now implementation targets, both at their leanest v1 shape: F1 ships zero new authoring surface (file + placeholder body only), and F5 ships the smallest thing that's still a real, valid pack skeleton (empty descriptor + stub build-check). tools/benchmark is being removed outright as part of this pass, not left dormant.

2 · What the Code Does Today

Both features attach to code that already exists:

  • For F1 — generate/markerScan.ts already scans AI_IMPLEMENTATION blocks and parses their optional pairing id + required flag; status/status.ts already re-scans pending records to decide resolved/unresolved; generate/report.ts already emits each block's file, line range, content, and required bit; generate/pendingTracker.ts already persists records across runs. F1 needs no new scanning — it's a new command that reshapes data these modules already produce.
  • For F5 — packages/templates-node/generic-v1 is the reference shape of a real (if minimal) pack; validate-pack already smoke-tests a pack directory against a descriptor; each pack ships its own tools/validate-build.mjs for the real toolchain build (see templates-dotnet/tools/validate-build.mjs). F5's MVP only needs to emit a schema-valid empty descriptor plus a stub of that build-check file — both are small, known shapes.
flowchart TD
  MS[markerScan.ts - finds blocks] --> ST[status.ts - resolved?]
  MS --> RP[report.ts - files/blocks]
  ST --> NEXT[scaffold next F1 - new]
  RP --> NEXT
  NODE[templates-node/generic-v1 - reference shape] -.pattern for.-> PACKNEW[scaffold pack new F5 - new]
  PACKNEW --> DESC[empty manifest.templates.json]
  PACKNEW --> STUB[validate-build.mjs stub]
  DESC --> VP[validate-pack]
