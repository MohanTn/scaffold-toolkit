1 · Overview — Feature, Audience & Dependencies

Feature Summary

Multi-Tenancy with Dual DbContext extends the v8-controller scaffold templates to generate projects with two separate Entity Framework DbContext classes: an ApplicationDbContext for shared, non-tenant-specific data (users, system settings, audit logs) and a configurable TenantDbContext (default name: RetailerDbContext) for tenant-scoped business data (invoices, products, orders). Entities are explicitly assigned to one context via a context field in the scaffolding manifest. This solves the problem of multi-tenant SaaS systems where shared and tenant-isolated data must be physically separated for security, compliance, and performance isolation.

Assumptions

  • Developers have basic familiarity with Entity Framework Core, DbContext, and EF migrations.
  • The scaffold CLI engine (@mohantn/scaffold-core) is already built and supports reading custom options fields in the manifest (confirmed: free-form options are passed to Handlebars context).
  • Two separate database connections/schemas are provisioned and their connection strings are injected at runtime (not hardcoded).
  • The TenantContext (configurable) can have the same or different provider (Sqlite, PostgreSQL, SQL Server) as ApplicationContext.
  • Repositories and handlers will be updated to accept and route queries to the correct DbContext (via constructor injection of both contexts).

Out of Scope

  • Tenant isolation enforcement: This design assumes connection string separation; row-level security (RLS) or query filters for isolation within a shared database are not covered (future enhancement).
  • Tenant provisioning / onboarding: Creating new tenants, database setup, schema migration per tenant — deferred to application code, not scaffold templates.
  • Multi-database failover / replication: Backup/restore and high-availability strategies are infrastructure decisions, not scaffold concerns.
  • Offline/sync scenarios: Replication and offline sync for mobile/edge clients out of scope.

Functional Requirements

ID Requirement Acceptance Criteria
FR1 Manifest configuration for multi-tenancy Manifest supports options.enableMultiTenancy: true/false and options.tenantContextName: "string" (default: "RetailerDbContext"). When enabled, scaffold generates two DbContext classes.
FR2 Per-entity context assignment Entity definition in manifest includes optional context: "Application" | "Tenant" field. If omitted, defaults to "Tenant". Scaffold generates entities in the assigned context's folder and DbSet collection.
FR3 Dual DbContext generation Templates generate ApplicationDbContext.cs (with DBSETS_APPLICATION marker) and {TenantContextName}DbContext.cs (with DBSETS_TENANT marker). Both inherit DbContext, apply EF configurations from assembly, and have separate connection string configuration.
FR4 Dual repository layers per context Scaffold generates repositories for tenant entities under Infrastructure/Persistence/Repositories and for application entities under a separate Application folder or shared Common. Each repository is injected with the correct DbContext via constructor DI.
FR5 DI registration for both contexts InfrastructureDependencyInjection.cs registers both AddDbContext<ApplicationDbContext>(...) and AddDbContext<{TenantContextName}>(...) with separate connection string keys: ConnectionStrings:Application and ConnectionStrings:Tenant.
FR6 Handler/Query routing to correct context Command/Query handlers for tenant entities receive I{Entity}Repository which is injected with the correct DbContext. Handlers for application entities use the application context. No handler code changes required; routing is transparent.
FR7 Configuration file support appsettings.json includes two connection string keys: ConnectionStrings:Application and ConnectionStrings:Tenant. Scaffold template generates both in the skeleton file.
FR8 EF Migration awareness Documentation and generated Readme.md (or comments in DependencyInjection.cs) notes that migrations must be created separately for each context using dotnet ef migrations add --context ApplicationDbContext and dotnet ef migrations add --context {TenantContextName}.

Non-Functional Requirements

NFR Metric Target Rationale
Performance Query latency (p95) <100ms (local)/<200ms (remote) per DbContext query Two separate connection pools ensure no cross-tenant contention; acceptable latency per best practice.
Security Data isolation Zero cross-tenant data leakage via connection string separation Separate databases/schemas + credentials prevent accidental queries to wrong context.
Availability Connection pool health Auto-recovery on DB restart; max 3 retries before circuit-break Resilience for production DB failover scenarios.
Compliance Data residency Tenant data in tenant DB region; application data in app DB region GDPR / data residency requirements satisfied via schema/DB separation.
Testability Unit test isolation SQLite in-memory DbContexts for both contexts in unit tests Fast, isolated tests without external DB dependencies.
Maintainability Template code duplication <5% duplicate code between application and tenant context paths Reuse helpers and shared components; minimize hardcoded context names.

Migration Impact & Notes

  • ApplicationDbContext migrations: Separate migration history from TenantDbContext. Create user-facing entities (User, Role) here. Run migrations with --context ApplicationDbContext.
  • TenantDbContext migrations: Separate migration folder. Create tenant-scoped entities (Invoice, Product) here. Run migrations with --context RetailerDbContext (or configured name).
  • No FK constraints across contexts: Foreign keys may reference users by ID (TenantDbContext.Invoice.CreatedByUserId) but the actual User object lives in ApplicationDbContext. Resolve via application code, not DB constraints.
  • Seed data: Application context seeds (users, roles) run once per app deployment. Tenant context seeds (demo products) run per-tenant during onboarding.
  • Connection string changes: If connection string changes at runtime (disaster recovery), both contexts reconnect independently; no schema synchronization needed.

DI Resolution for Dual Contexts

  • Service resolution: When a handler requests IInvoiceRepository, the DI container retrieves the singleton instance bound to that interface. That instance was constructed with TenantDbContext (because Invoice is a tenant entity).
  • Context lifetime: Each DbContext is scoped to the request (created at request start, disposed at request end). This ensures connection pool reuse and proper async tracking.
  • Connection pool management: Each DbContext maintains its own connection pool (connection string A for ApplicationDbContext, connection string B for TenantDbContext). No cross-contamination.
  • Transaction boundaries: If a handler must write to both contexts, it uses TransactionScope or manual distributed transaction logic (complex; usually avoided by keeping writes to one context per request).

Risks & Mitigation

Risk Likelihood Impact Mitigation
Data cross-contamination: Developer accidentally queries application data in tenant context or vice versa. Medium Critical (data breach, regulatory fine) 1) Separate connection strings enforce isolation at DB layer. 2) Code review + tests. 3) Generated comments in handlers warn about context selection. 4) Runtime logging tags queries by context for audit.
Migration chaos: Migrations for two contexts get out of sync; one context lags in schema updates. Medium High (data corruption, app crashes) 1) Generated Readme documents the two-context migration pattern. 2) CI/CD pipeline runs both dotnet ef database update commands in sequence with explicit --context flags. 3) Monitoring alerts on migration failures.
Connection pool exhaustion: One DbContext's pool runs out of connections, blocking all queries. Low (cloud-managed DB auto-scales) High (API unavailability) 1) Each DbContext has independent connection pool (not shared). 2) Alerts on pool saturation (>80% used). 3) Kubernetes pod autoscaling on latency percentile.
Scaffold template bugs on first pass: Generated code has syntax errors or missing imports. High (new feature) High (broken generated project) 1) Comprehensive Mermaid test of template generation against the example Invoice + User entities before release. 2) Unit tests in scaffold CLI (if applicable). 3) Early community feedback on generated output.
Deployment downtime during dual-DB migration: One DB migrates faster than the other; app tries to use new schema against old DB momentarily. Medium High (transient API 500 errors) 1) Migrations are backward compatible (add column before use). 2) Deployment waits for both contexts to complete before health check. 3) Canary/blue-green deployment on staging before prod.
Tenant context naming collision: User sets tenantContextName: "ApplicationDbContext", causing duplicate class names. Low Medium (compilation failure, confusing error) 1) Scaffold CLI validates that tenantContextName != "ApplicationDbContext" and raises early error. 2) Manifest schema includes regex constraint on the name.
