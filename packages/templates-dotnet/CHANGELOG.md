# Changelog

All notable changes to this project are documented here. The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/), and this project adheres to [Semantic Versioning](https://semver.org/).

## [Unreleased]

### Added

- v8-controller: brownfield parity. The pack now declares its own `bootstrapAnchors` (fixing the built-in catalog's stale `AppDbContext.cs`/`ApplicationServiceCollectionExtensions.cs` filenames â€” the pack's real files are `ApplicationDbContext.cs`/`DependencyInjection.cs`). The `di-registration` snippet uses fully-qualified static invocations so the host `Program.cs` needs no `using` directives. A new `DevSchemaBootstrap` hosted service (Development-only `EnsureCreated`, registered by `AddInfrastructure`) replaces the Program.cs schema-bootstrap block, and a new `ApiExceptionFilter` (registered via `AddApi`'s `AddControllers` options) maps `ValidationException`/`NotFoundException` to 400/404 independent of the host's middleware pipeline. A generated `Directory.Build.targets` wires an existing brownfield `*.Api` project to the module's Application/Infrastructure projects without editing its `.csproj` (the generated API project opts out via `ScaffoldModuleProject`).
- v8-controller: each generated `.csproj` now carries its own `TargetFramework`/`ImplicitUsings`/`Nullable` properties; the repo-root `Directory.Build.props` target is gone (it silently applied to every project in a brownfield repo).

- v8-controller: foreign-key relationship support in the intent manifest. A field with `references: "<Entity>"` now renders a nullable navigation property on the entity, and the EF `HasOne(...).WithMany().HasForeignKey(...)` mapping with a configurable `onDelete` behavior (default Cascade). New `examples/basic-user.manifest.json` + `examples/related-order.manifest.json` demonstrate the two-entity flow, and `EntityConfiguration.cs.hbs` gains an `AI_IMPLEMENTATION` seam for extra mapping (indexes, value conversions) so it survives as a marked edit surface.
- v8-controller: `isNullable` pack helper; nullable field types (`string?`, `int?`) no longer get `.IsRequired()` in the EF configuration.

### Fixed

- v8-controller: removed a stray never-injected `REPOSITORIES` marker pair from `ApplicationDependencyInjection.cs.hbs` (it duplicated the real one in the Infrastructure file and broke marker-based file disambiguation).
- v8-controller: `companyProjectName`/`ns` helpers honor a root-level `companyProjectName` (persisted in the pack slot by `scaffold bootstrap-markers` or set in the manifest), so existing single-token layouts like `ThetaDesk` are representable.
- v8-controller: every POST returned 500 (entity still persisted) because `EntityController.cs.hbs` used `CreatedAtAction(nameof(GetByIdAsync), new { id = result }, ...)` â€” MVC trims the `Async` suffix from action names and the whole DTO was passed as the route `id`, so route generation threw `InvalidOperationException`. Now `CreatedAtAction("GetById", new { id = result.Id }, result)`.
- v8-controller: `EntityDto.cs.hbs` omitted `Id`, so API clients could never learn a created entity's id â€” which made any foreign-key workflow impossible over HTTP. DTOs now include all manifest fields, and the four handler templates map `Id` through.
- v8-controller: generated apps had no database schema bootstrap; every request failed with "no such table" until `Program.cs` was hand-edited. Schema now bootstraps via the `DevSchemaBootstrap` hosted service in Development (migrations remain the production path).
- v8-controller: Create/Update handler placeholders now map all manifest fields from the request instead of shipping an empty entity, and the five CRUD handler seams dropped the `:required` tag â€” their placeholders are complete implementations, so `scaffold status` is green immediately after generate instead of demanding no-op edits.
- v8-controller: string entity properties initialize to `string.Empty` (silences CS8618), the EF configuration no longer emits a duplicate `Id` property line, and `Microsoft.NET.Test.Sdk`/`Swashbuckle.AspNetCore` pins moved to versions that exist on nuget.org (17.9.0 / 6.5.0), eliminating NU1603 restore warnings.

- This change consolidates the template library by removing support for v10-minimal-api (both standard and GCP variants) and v8-controller-gcp variants, while significantly expanding the v8-controller template pack from base templates to 67 comprehensive templates covering CQRS patterns, domain-driven design, and multi-tenancy. A new architecture document establishes the dual-DbContext multi-tenancy pattern, the rendering engine gains pathConfig support for dynamic output paths, and configuration examples are enhanced with database provider and company/project metadata fields.
- This change introduces template pack validation automation. A new `validate-all.mjs` script auto-discovers all template packs and runs the existing smoke-test renderer against each using a shared example manifest. A GitHub Actions workflow runs this validation on every push to main and pull request across Node 18, 20, and 22. Package-lock.json is now committed for reproducible builds, and documentation updated to reflect the new npm test entry point.
