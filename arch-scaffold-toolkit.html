🏛️ Architecture — Scaffold Toolkit Architecture

End-to-end architecture of the @mohantn/scaffold-toolkit monorepo: a deterministic, LLM-agnostic Node CLI (scaffold-core) renders Handlebars templates and splices marker-based register snippets into existing files, while two thin host-agent adapters (Claude Code and GitHub Copilot CLI) wrap that CLI in a three-layer enforcement surface (PreToolUse → PostToolUse → Stop / agentStop, plus per-invocation scaffold-status prechecks) that keeps AIs on the business logic while templates own the boilerplate. · TypeScript · Node.js ≥ 20.12 · Handlebars · AJV · Commander · TOON

🤖 AI Overview — What Will Be Implemented

This document captures how the scaffold-toolkit monorepo actually works, end to end.

  • Portable engine — @mohantn/scaffold-core, the Node CLI scaffold, owns every byte a template can know about: entity/domain objects, DTOs, controllers, query/command handlers, validators, DI registration, marker scaffolding, and unit-test shells.
  • Host enforcement surface — two thin adapters (the Claude Code Skill + hooks, and the GitHub Copilot CLI gh-scaffold + hooks) wrap that CLI in a structural gate so the agent can never author boilerplate it does not own; the only legal edit an agent makes lives inside an AI_IMPLEMENTATION:START/END interior.
  • Reference template packs — packages/templates-dotnet (versions v8-controller and v8-controller-gcp) and packages/templates-node/generic-v1 ship as in-repo source-of-truth and prove the engine is fully stack-agnostic by real-building each test_data/ fixture in CI.
  • Benchmark + publish — a reusable Claude-API benchmark arms compares scaffolded vs. freehand circles, and a release bot publishes @mohantn/scaffold-core + @mohantn/scaffold-adapter-copilot-cli to the npm public registry on every successful main merge, gated by both build-check jobs.

Sections 1–9 each cover their own concern (overview, use cases, C4, data model, API, developer experience, behaviour, deployment, monitoring). Section 10 lists every architectural commitment made so future contributors can see and challenge them before proposing changes.

1 · Overview — Feature, Audience & Dependencies

Repository

The scaffold-toolkit monorepo is a monorepo containing:

  • packages/core — @mohantn/scaffold-core (published, 0.2.8): the portable Node CLI scaffold and its runtime library.
  • packages/adapter-claude-code (source-only, private: true): the Claude Code Skill scaffold plus the four hook scripts that wire the engine into a Claude Code session.
  • packages/adapter-copilot-cli — @mohantn/scaffold-adapter-copilot-cli (published): the GitHub Copilot CLI gh-scaffold subcommand dispatch + its three hooks + the install-hooks writer.
  • packages/templates-dotnet (source-only, never built/linted/tested as a workspace): two reference versions v8-controller and v8-controller-gcp, each with its own test_data/ fixtures, tools/validate-build.mjs, and tools/check-guardrails.mjs.
  • packages/templates-node (source-only): generic-v1 — a Node.js/TypeScript module scaffold (Module + Registry + tests + tsconfig + package.json + registry-link), proving the engine is stack-agnostic.
  • tools/benchmark/: the reusable scaffolded-vs-freehand comparator used to remeasure ROI across refactor rounds.

It exists to keep LLM-driven code generation deterministic: the CLI owns every byte a template can know about (entities, DTOs, controllers, DI registration, query handlers, validators, tests, AppDbContext registration); the host agent — via its adapter — owns only the business-logic seams the template leaves inside AI_IMPLEMENTATION:START/END markers, and is structurally forbidden from authoring the rest.

Non-functional requirements

AttributeMeasurable target
DeterminismSame manifest + same pack + unchanged working tree → byte-identical generator output and identical report for every repeat run, except for the hash trailer comments
Sub-second typical generateThe v8-controller reference pack renders, injects, scans, and writes in well under 1 second on a developer laptop (the dotnet pack's Invoice fixture completes inside 5s end-to-end including a real dotnet build in CI)
No networkThe CLI itself never reaches the network. The only network-touching piece in this monorepo is the benchmark harness's Claude API call
Hallucination-proof boilerplateA host agent cannot inject boilerplate (DTOs, controllers, DI registration) because PreToolUse structurally refuses; the only legal agent edit lives inside an AI_IMPLEMENTATION interior and is the only edit PreToolUse allows
Stack-agnosticismThe same engine renders .NET (v8-controller, v8-controller-gcp) and Node/TS (generic-v1) without any stack-specific branch in core; stack concerns live entirely in pack descriptors and templates
Local-only stateEvery byte is under the target repo's .scaffold/; deleting .scaffold/ is the safe rollback (config still loses its provenance and so on, but the working code is intact)
Idempotent re-generatescaffold generate on an already-generated tree never duplicates injection content (replace strategy: prior injection blocks are recognized by hashTrailerPrefix; append strategy: identical snippets collapse)
AuditabilityEvery generate writes a changeset manifest before any file is touched, so the run is reversible byte-for-byte through scaffold undo

Migration impact

None. The .scaffold/ directory is a local-only, ephemeral rebuild-on-demand state; the engine treats it as if it might be deleted at any time. The only stable invariant across re-generations is .scaffold/config.json, which records pack ownership and provenance hooks per file; everything else is computed again from the manifest + the current pack descriptors + the current working tree. There are no migrations, no version-up globs, no upgrade hooks — deleting .scaffold/ (except .scaffold/config.json) is the explicit recovery path any time a build-check is wedged on stale pending state.

The intent-manifest (manifest/schema.ts) and the descriptor (descriptor/schema.ts) each carry their own ajv version field (manifestSchemaVersion: 1, descriptorSchemaVersion: 2). Adding a new top-level field is additive; renaming a top-level field would require a published schema bump and a migration story, neither of which has happened.

Honest scope note

This repo is shipped as a build/CI surface, not a production service; failures show up as red PRs and stopped pipelines, not as user impact. There are no live production deployments to monitor in the traditional sense, and no PII or telemetry from consumer repos. The metric “SLOs” above are therefore advisory: they describe what we want the build to look like, not what users will feel.
