🏛️ Architecture — Generic Template Pack: the agent fills placeholders, never authors files

1 · Overview — Feature, Audience & Dependencies

Feature

Generic Template Pack generalizes the proven v8-controller .NET pack into a stack-neutral authoring format on the existing scaffold-core engine. The engine is already pack-driven across languages (descriptor v2 carries inputs, commentSyntax, bootstrapAnchors); this feature (a) ships a non-.NET reference pack that proves it, and (b) closes the residual assumptions in core that still lean on the .NET pack. The governing contract is zero file authoring by the agent: for any new resource the agent emits a one-line spec and, after scaffold generate writes the whole file tree and wires it together, edits only the SCAFFOLD:AI_IMPLEMENTATION interiors. Everything else is deterministic, so per-resource output tokens collapse to the handful of lines that genuinely need reasoning.

Assumptions

  • Descriptor schema stays at v2; this feature is additive and needs no schema bump — inputs, commentSyntax, and pack-local helpers.js already exist.
  • The host runs behind a PreToolUse hook wired to scaffold check-edit; without it, the contract is advisory, not enforced.
  • Packs are consumed as local directories (--pack name=path@version); init no longer accepts git URLs.
  • One pack = one language/stack; a multi-language single pack is out of scope (see Section 10).

Out of scope

  • A single pack spanning multiple languages in one descriptor.
  • Executing arbitrary code in helpers (pure string transforms only).
  • Re-generating or editing existing business logic outside the holes.
  • Git-URL / remote pack fetching in init.

Functional requirements

IDRequirementAcceptance criteria
FR1A pack declares its own input vocabulary via inputs[]; the engine enforces it without the legacy .NET entity+fields contract.Generic pack with inputs: [{name, type, required}] validates a manifest that has no entity; an existing dotnet pack (no inputs) still enforces PascalCase entity + non-empty fields.
FR2The generic pack renders arbitrary file types from targets[] deterministically.Two consecutive generate runs on the same manifest produce byte-identical files; extensions outside the built-in table render because the pack supplies commentSyntax.
FR3Cross-file linking is expressed as an injections[] append into a registry/index file.Scaffolding a second resource appends one link line to the registry marker; re-running is an idempotent no-op guarded by the hash trailer.
FR4Each generated file exposes exactly the minimal AI_IMPLEMENTATION holes; everything else is frozen.scaffold status lists the holes; the placeholder already compiles; a :required hole stays tracked until its content changes.
FR5check-edit blocks any write to a pack-owned file and any edit outside a hole interior.check-guardrails.mjs against the generic pack: write of a whole target blocked, edit of a link marker blocked, edit inside a hole allowed.
FR6The generic pack ships test_data/ and a build-gate that compiles/tests its output in CI, gating publish.A new CI job builds the scaffolded sample (tsc + node --test) and sits in the publish job's needs: list.
FR7Marker comment syntax for the generic pack's extensions resolves from the pack's commentSyntax map.Injection markers render as valid comments for .ts without any change to core's built-in TABLE.

Non-functional requirements

AttributeMeasurable target
Token efficiency≥90% of per-resource output tokens are deterministic (rendered), agent emits only spec + hole bodies.
DeterminismRe-render of an unchanged manifest is byte-identical (0 diff).
Stack decouplingGeneric pack builds green in CI with only Node + tsc; zero .NET SDK dependency.
Fail-closed safetyAmbiguous / not-found / multi-match old_string is blocked, never guessed.
CI gate latencyNode build-check completes in <90s so it does not dominate the pipeline.

Migration impact

No schema migration. Descriptor stays descriptorSchemaVersion: 2 and intent manifests stay manifestSchemaVersion: 1. The only core change is generalizing manifest/inputValidation.ts so a pack that declares inputs is validated against its own vocabulary rather than the hard-coded entity+fields legacy path (already the documented behavior of inputs). Existing dotnet manifests and generated trees are unaffected — the legacy branch is retained for descriptors that omit inputs.

    // SCAFFOLD:AI_IMPLEMENTATION:END
